Personal Agent Harness · Design Spec · v1.4

Wrapping Claude Code into a persistent agent — APPROACH

A self-hosted harness that turns Claude Code from a tool you invoke into a process that persists, acts on its own schedule, and curates its own memory — with its own thin channel adapters, running on the homelab box behind Tailscale. This spec sizes exactly how much of that you already have, and marks the part that is real work.

§1

Purpose & scope

Reconstruct the useful properties of OpenClaw / Hermes on infrastructure you own and understand, so the "always-on, self-improving assistant" behavior isn't rented from a framework whose harness you can't see into.

Goals

  • Persistence. A message, a webhook, or a timer continues the same live session hours later — not a fresh stateless invocation.
  • Proactivity. The agent wakes itself on a schedule, decides whether anything is worth surfacing, and stays quiet when it isn't.
  • Accretion. Facts and successful patterns persist across sessions and are pulled back in on relevance — the agent's competence compounds.
  • Ownership. Everything runs as plain files, SQLite, and systemd units on your box, behind Tailscale, with no vendor control plane.
  • Orchestration. Multi-session work — including coding-agent work across repos — lives in a git-native task graph (beads). Ready-work detection decides what gets picked up; blocked work parks with its blocker instead of rotting in prompts or leaking into memory.

Non-goals

  • Matching OpenClaw's 20+ hardened channel adapters. You wire the two or three surfaces you actually use.
  • A polished install wizard or GUI. This is a homelab system for one operator.
  • Multi-tenant / SaaS shape. That's a separate decision (the BYOK angle you've mulled), explicitly out of scope here.
§2

The gap, precisely

Claude Code + skills already covers the expensive-to-build part: the reasoning engine and its tool loop. The delta is the harness around it — channels, persistence, scheduling, and a curated memory loop.

What the engine already gives you (P0 — installed)

  • Headless agent loop. claude -p "…" runs the full tool-using loop and exits; --output-format stream-json gives you an event stream the daemon relays out to channels.
  • Session resume. Pin a UUID at spawn with --session-id, then --resume <id> on every later turn (capture from --output-format json also exists, but the harness pre-pins — §4.1). This is the primitive the whole persistence layer is built on — you don't reimplement conversation state, you index into Claude Code's.
  • Deterministic hooks. A ~30-event lifecycle (SessionStart, UserPromptSubmit, PreToolUse, PostToolUse, Stop, SessionEnd, PreCompact …) that fire regardless of model cooperation. This is your enforcement + reflection substrate.
  • MCP + skills. .mcp.json servers and .claude/skills/*/SKILL.md — plain-file skills that live in the workspace and travel with git.
  • A memory head-start. Claude Code already persists context under ~/.claude/projects/<proj>/memory/ with a MEMORY.md index that auto-splits into topic files. It's rudimentary, but it means you're extending an existing store, not starting from zero.

What's missing (the build)

  • Channel adapters — thin, native connectors (discordgo, slack-go) for the 2–3 surfaces you actually use, normalizing every entry point into the §6 event contract and stamping trust/thread_key/dedup_key in code you own. Small per surface. (cc-connect was evaluated for this role and rejected — §10.)
  • Session router + scheduler — mapping channel threads to sessions, and a heartbeat that invokes the agent unprompted. Thin-to-small; mostly plumbing over primitives that already exist.
  • Curated memory + recall — agent-authored facts, relevance retrieval (FTS5), and log compaction that you control rather than accept as a black box. Moderate.
  • Self-improvement loop — reflection that extracts durable facts and synthesizes new skills from successful runs. The genuinely hard part is the synthesis half — deliberately deferred (§10): v1 ships fact extraction and rollups; skill synthesis waits on an evidence trigger.
  • Policy + sandbox — because inbound messages and fetched web content are untrusted, this is not optional. Config-heavy but must-do.
Read this as a build map with one decided fork. Sections §3–§8 spec the whole system; §9 sequences it; §10 records the build-vs-buy decision: build it, defer skill synthesis (F2) behind an evidence trigger — the Hermes split was evaluated and rejected on structural grounds.
§3

System architecture

Five layers. One you already run (engine), four you build (channels, APPROACH harness, memory, policy) — though the channel layer is thin. APPROACH is one small resident daemon that owns its channel adapters and sequences events onto the engine — approach control handing traffic to the runway.

Shape decision. APPROACH is a single small Go daemon — channel adapters included — not a per-event dispatcher. Four requirements force it: per-thread serialization (a lock that must span processes), live stream-json relay, durable pending approvals, and a durable event queue written on receipt (§4.1). One resident process with a per-thread goroutine + queue over the events table makes all four trivial; systemd timers shrink to socket pokes, and the daemon spawns each session from its recorded cwd (§6) — which also closes the --resume cwd-scope trap (§11).
§4

Control flows

4.1  Inbound message

The hot path. First act on receipt: persist the event row (§6) — gateway channels don't redeliver, so receipt is the last moment durability is free; everything after is recoverable from the table. The engine never invents a session — the router resolves it from the store — and the Stop hook records the turn asynchronously so the reply isn't blocked on memory writes. Queues are keyed by thread_key, not session_id, and claimed before resolution: two simultaneous first-messages to a new thread serialize in the queue, and the sessions schema's partial unique index (§6) backstops the one-active-session-per-thread invariant even against daemon bugs. In-memory queues are just an index over unprocessed event rows — per-thread FIFO by receipt order, rebuilt from the table on restart. The reply leg is at-least-once: the event advances to replied only after the channel adapter confirms the platform accepted the send, so a crash in that window re-sends on restart — a rare duplicated chat message is accepted; a silently eaten reply is not.

4.2  Heartbeat (proactivity)

A timer, not a message, drives this. The agent reads a checklist and exercises judgment on a schedule; a no-op result is swallowed so silence is the default. Run it on a cheap model (Haiku) to keep idle cost near zero. Three protocol decisions: heartbeats run in a fresh session every fire (a persistent heartbeat session accumulates cruft and cost); "something to say" is expressed as a notify tool call, never a sentinel string parsed out of free text — no tool call means silence; and the prompt reads/writes a notifications ledger (item_key, notified_at, snooze_until) so a 30-minute timer doesn't re-nag the same deadline 16 times a day. NL-cron ("brief me at 8am") is a schedules row the agent inserts via a skill — the scheduler fires rows, not just HEARTBEAT.md. Every fire has a manufactured identity — (schedule_id, occurrence_time) is the event's idempotency key — and misfires coalesce with a staleness flag: after downtime, at most one catch-up event fires per schedule, keyed to the newest missed occurrence and carrying missed_count + lateness so the model judges whether a 3-hour-late brief still ships; older occurrences are recorded as skipped, never enqueued. Three more contract points. Trust: system-generated events (heartbeat, cron, reflection triggers) carry trust=system — stampable only by the daemon itself, never by an adapter; it reads like owner for reads, notify, and task-graph operations, but can never satisfy an approval, and its sends are restricted to configured targets. Time: each schedule stores an IANA timezone; cron expressions are DST-aware local wall-clock (a nonexistent 2:30am fires at the next valid instant; a repeated one fires once), and occurrences are recorded UTC + zone. Overlap: max_overlap = 1 by default — an occurrence due while the previous fire's session still runs is recorded as a skipped event row, never enqueued; fires and skips alike are event rows, so the per-fire audit trail is the events table itself, and a chronically-skipping schedule shows up in the digest as a tuning signal. One standing checklist item: bd ready — unblocked task-graph work is surfaced (or picked up, §4.5) through the same judgment-on-a-schedule loop. The heartbeat is the only component that asks beads what's ready; beads never initiates.

4.3  Context assembly

What actually goes into each turn. Keep it lean: the model gets a skills list (names + descriptions, not bodies) and only the facts that retrieve against the current input — this is where a black-box harness quietly bloats your token bill. Mechanism is pinned: the UserPromptSubmit hook, not a prompt prefix. And because --resume already carries the session's history, the full fact package is injected only on session start/rotation; per-turn injection is the retrieval delta — facts matching this input that aren't already in-session. "Already in-session" is a ledger, not a guess: the assembler records every injected fact id in session_facts (§6), and each turn injects FTS hits minus that set — ranked by match score, confidence, and recency, capped by a top-k / token budget in approach.toml (start k=8). Re-paying the whole fact block every turn is exactly the bloat this figure warns about.

4.4  Approval round-trip (gated verbs)

The hardest flow in the system, so it's pinned: deny-and-resume. Hooks are synchronous with timeouts — the gate never holds a turn open waiting for a human. It records the intended call, denies with a reason, and lets the turn end gracefully; your approval arrives later as just another inbound event that resumes the session. Approvals are durable rows, so they survive restarts, and a gated call can never fire without a matching approved row. The binding is cryptographic, not textual: the row stores the canonical tool name and a digest of the normalized JSON arguments (the rendered string is display-only), plus the requesting event id, session, and timestamps; the approval id doubles as the nonce. Authorization: an approval event is valid iff its sender resolves to the owner identity on a trust=owner surface — any owner surface, not just the requesting one (approve from Slack what Discord asked; it's the same authenticated human). Expiry: 1 hour — a stale "approve 42" can't fire tomorrow against forgotten context; expired means a fresh request cycle. Match-and-consume is one transaction; if the resumed model re-issues a call whose digest differs from what was approved, that is a new approval cycle, never a fuzzy match.

4.5  Task orchestration (coding-agent work)

Multi-session work lives in beads (C13), not in prompts. A ready issue becomes a worker session: a row in the same sessions table, keyed task:<issue-id>, spawned from the repo's cwd (stored per session — §6; this is what forces the sessions.cwd column, since --resume is scoped to the spawn directory). Workers run under the same policy matrix and the same per-thread queue; a worker surviving a reboot is just C4 persistence applied to work instead of conversation. Done → bd close with a closing note the reflector (§5) can read; stuck → bd update --blocked-on, and the issue drops out of ready until its blocker clears — parked, not nagging, not leaking into facts.

4.6  Failure & recovery

One rule unifies every failure path: retry only what provably did nothing; surface everything else. Events are never silently dropped because every path below ends in a durable state a human can see.

  • Daemon crash mid-turn. On restart, events found in processing become interrupted — never auto-rerun. The side effects of a half-finished turn are mechanically unknowable (the email may or may not have gone out), so the daemon posts to the originating thread: what it was doing, and an offer to retry. Interrupted events park out-of-band — the thread's queue keeps flowing; a "retry" re-enqueues the old event at the tail, executing against current session state (--resume re-reads the transcript, which is exactly what absorbs the staleness). A frozen assistant that won't answer "hello?" until you adjudicate a crash report would be the worse failure.
  • Engine failure, daemon alive (non-zero exit, timeout, rate limit). Here the daemon has what the crash case lacks: live PostToolUse accounting for the turn. If zero side-effecting tool calls were observed: auto-retry, up to 2 attempts with backoff, honoring rate-limit signals. If any side-effecting call fired before the failure: no retry — same interrupted + surface path as a crash. Timeouts kill the child, then apply the same test.
  • Reply undelivered. Turn completed but the platform send was never confirmed (§4.1): re-send on restart. At-least-once, duplicate accepted.
  • Resume failure. The pinned session UUID points into Claude Code's store under ~/.claude/projects/; if that transcript is gone or unreachable (GC, corruption, moved cwd — §11), the row is marked resume_failed (a rotation cause, kept for forensics), a fresh session is created and seeded with the assembler's full fact package, the event processes there, and the reply carries a one-line note: "lost this thread's conversation history — facts intact, starting fresh." Transcript loss degrades to amnesia-with-notes, not an error; facts in approach.db are the durable memory by design. (Backing up the Claude-side store is a §6 ownership question.)
  • Dead letters. Terminal failures land in dead_letters (§6): retry budget exhausted, malformed or unroutable events, and interrupted events whose surface attempt itself failed. Exit is two independent paths — one owner-DM notification on entry (through the normal outbound leg), plus a standing heartbeat checklist item for unacknowledged rows (via the notifications ledger, so no re-nag). Draining is deliberately manual: a requeue/discard command per row. "Dead" means the machine has given up and a human decides.
§5

Memory & the learning loop

This is the 30% within the 30% — the part that separates "Claude Code with a cron job" from something that genuinely improves. Worth being honest about why it's hard.

Why it's the hard part

  • Fact curation is lossy and adversarial to itself. Naive "append what you learned" produces contradictory, duplicated, stale facts within days. You need dedup/merge, supersession ("X moved from A to B"), and decay — a small ontology problem, not a prompt. The facts table in §6 makes these row operations instead of markdown surgery, which is why SQLite (not MEMORY.md) holds the truth. V1 posture: the reflector auto-writes facts with provenance, confidence, and supersession links; decay is passive (stale facts rank down in retrieval, never auto-delete); a periodic owner-DM digest lists new/merged/superseded facts for one-tap pruning — human oversight without a blocking queue. Facts derived from trajectories containing untrusted content are stored immediately but excluded from context assembly until acknowledged in a digest — the trust split applied at read time (§7, §11).
  • Skill synthesis needs a taste filter — which is why it's deferred. Auto-generating a SKILL.md from one lucky run yields brittle, over-fit skills. The honest version watches for repeated success, drafts, and puts a review gate in front of anything that lands in the skill dir — an open-ended tuning project, deferred behind the evidence trigger in §10. The gate's cheap concrete mechanism, when it lands, is cross-model critique: Claude drafts the skill, codex exec (C12) attacks it with a different model lineage — de-correlated errors are exactly what a same-family self-review can't give you — and you approve. C12 ships at launch as an on-demand critic, so the mechanism is proven before F2 needs it.
  • Retrieval quality caps everything. Start with SQLite FTS5 — fast, transparent, no embedding infra. Add vectors only if lexical recall provably misses. Resist the urge to over-build here first.
  • Compaction is where memory rots silently. Summarizing yesterday's logs discards detail you can't get back. log_entries rows are append-only; compaction writes additional rollup rows (rendered into rollups/), never destructively — the raw rows survive every summary pass.
  • Reflection is a job, not a hook side effect. Every run is a reflection_jobs row (§6) under a single-holder lease with a durable watermark over log_entries + closed sessions. The nightly sweep is the correctness guarantee — it reflects everything from the last watermark to now, so nothing is ever missed; Stop/SessionEnd triggers are opportunistic freshness enqueues of the same job kinds, deduped by the unique (kind, range) key — nice when they run, harmless when the daemon dies (tonight's sweep covers it). A fact merge is one transaction: insert the merged fact, set superseded_by on the olds, copy fact_sources rows, append the digest entry — no partially-merged state is ever visible.
Decision hinge — resolved. C4/C5/C6 are a weekend or two of plumbing over primitives you already have. Within C8, the open-ended part is F2 skill synthesis; F1 fact curation is bounded by the v1 posture above (auto-write + digest + append-only means curation errors are recoverable, not compounding). §10 records the decision: build F1/F3 now, defer F2 behind an evidence trigger, Hermes rejected.
§6

Data model & layout

Everything is plain files + one SQLite db — greppable, git-diffable, recoverable.

Source-of-truth rules. Every piece of structured state has exactly one owner — facts living in both a markdown file and an index is the classic setup for silent divergence. So: SQLite is truth; MEMORY.md is a rendered view, regenerated after each reflection pass and committed to git. Dedup, supersession, decay, and provenance are row operations; the greppable/diffable goal survives because you diff the render. A second truth model exists on purpose — beads' git-synced JSONL — kept honest by a strict noun partition: facts = knowledge (approach.db), beads = work. If a row could be phrased as "do X," it's an issue, not a fact; if it's true after the work is done, it's a fact, not an issue.

Inbound event schema (C1 → C4 contract)

The daemon's channel-adapter layer (C1) normalizes every entry point — message, heartbeat, webhook, cron, approval — into one shape. The trust field is load-bearing: policy (§7) and sandbox routing key off it, and it is assigned by the adapter from channel identity — code you own by construction (§10) — never inferred from content.

State store (SQLite)

CREATE TABLE events (                -- THE durable queue (§4.1) — written on receipt, before processing
  id INTEGER PRIMARY KEY,
  dedup_key TEXT NOT NULL UNIQUE,    -- identity per kind (see contract above) — dup insert = no-op
  thread_key TEXT NOT NULL, kind TEXT NOT NULL, trust TEXT NOT NULL,
  payload TEXT NOT NULL,             -- full normalized event JSON
  status TEXT DEFAULT 'received',    -- received | processing | completed | replied
                                     --   | interrupted (parked, human decides — §4.6)
                                     --   | skipped (coalesced misfire — §4.2) | dead
  attempts INTEGER DEFAULT 0,        -- side-effect-aware retry budget (§4.6): max 2, only if
                                     --   zero side-effecting tool calls observed this turn
  received INTEGER NOT NULL, updated INTEGER,
  correlation TEXT                   -- links retries / approval round-trips to the origin event
);
CREATE INDEX ev_queue ON events(thread_key, id) WHERE status IN ('received','processing');

CREATE TABLE dead_letters (          -- terminal failures, surfaced twice (§4.6): DM on entry
  event_id INTEGER PRIMARY KEY REFERENCES events(id),
  reason TEXT NOT NULL,              -- retries-exhausted | malformed | unroutable | surface-failed
  entered INTEGER NOT NULL,
  acked INTEGER,                     -- owner saw it (heartbeat re-surfaces unacked rows)
  resolution TEXT                    -- requeued | discarded — manual, per row
);

CREATE TABLE sessions (
  thread_key   TEXT NOT NULL,        -- per-channel contract above (incl. task:<issue-id> workers)
  session_id   TEXT NOT NULL,        -- UUID generated BY THE DAEMON, spawned via --session-id —
                                     --   never captured from engine output (§4.1)
  status       TEXT DEFAULT 'creating',
                                     -- creating → active → rotated (on /new, idle TTL, turn cap)
                                     --   | resume_failed (transcript lost — rotation cause, §4.6)
                                     --   | failed (creating past deadline — retried fresh)
  cwd          TEXT NOT NULL,        -- spawn/resume dir: ~/approach, or repo path for task:* workers
                                     -- (--resume is scoped to the spawn dir — assert at spawn)
  tainted      INTEGER DEFAULT 0,    -- sticky (§7): set by PostToolUse on untrusted bytes
                                     --   (web fetch · non-owner participant · Codex output);
                                     --   mutating verbs read one policy column right; clears on rotation
  channel      TEXT, last_seen INTEGER, turns INTEGER,
  PRIMARY KEY (thread_key, session_id)
);
-- THE router invariant, enforced in schema: at most one live session per thread.
-- Rotation = one transaction (old→rotated + new→creating); any bad interleaving
-- is a constraint violation, not a race.
CREATE UNIQUE INDEX one_live_session ON sessions(thread_key)
  WHERE status IN ('creating','active');

CREATE TABLE facts (                 -- SOURCE OF TRUTH; MEMORY.md is rendered from this
  id INTEGER PRIMARY KEY,
  fact TEXT NOT NULL,
  provenance TEXT NOT NULL,          -- 'owner' | 'agent-reflection' | 'untrusted-derived'
                                     --   (untrusted-derived: excluded from assembly until digest-ack — §5)
  confidence REAL DEFAULT 0.7,       -- reflector-assigned; decays passively via ranking, never deletes
  created INTEGER,
  superseded_by INTEGER REFERENCES facts(id),  -- non-destructive supersession
  confirmed_count INTEGER DEFAULT 1, -- cache; derived from fact_sources
  last_used INTEGER
);
CREATE TABLE fact_sources (          -- confirmations are ROWS, not an overwritten column
  fact_id INTEGER NOT NULL REFERENCES facts(id),
  source_ref TEXT NOT NULL,          -- thread_key, log_entries id, or event id it came from
  trust TEXT NOT NULL,               -- trust of the source at extraction time
  ts INTEGER NOT NULL
);

CREATE TABLE log_entries (           -- SOURCE OF TRUTH for logs; daily .md is rendered from this
  id INTEGER PRIMARY KEY,            -- append-only: written by the Stop hook path
  ts INTEGER NOT NULL, session_id TEXT, kind TEXT,  -- turn | note | rollup
  text TEXT NOT NULL
);

CREATE TABLE reflection_jobs (       -- §5: durable, leased, watermarked
  id INTEGER PRIMARY KEY,
  kind TEXT NOT NULL,                -- extract-facts | compact | render
  source_from INTEGER, source_to INTEGER,  -- log_entries id range (the watermark advance)
  lease_holder TEXT, lease_expires INTEGER,-- single-holder; expired lease = takeable
  status TEXT DEFAULT 'queued',      -- queued | running | done | failed
  attempts INTEGER DEFAULT 0, output TEXT,
  UNIQUE (kind, source_from, source_to)    -- opportunistic + nightly enqueues dedup here
);

CREATE TABLE session_facts (         -- retrieval-delta ledger (§4.3): what this session has seen
  session_id TEXT NOT NULL, fact_id INTEGER NOT NULL, turn INTEGER,
  PRIMARY KEY (session_id, fact_id)
);

-- lexical recall: two external-content FTS tables, trigger-synced to their
-- source tables; `approach reindex` rebuilds both from scratch

);

-- overlap/misfire skip) is an event row keyed (schedule_id, occurrence_time)

  id INTEGER PRIMARY KEY,

  input_tokens INTEGER, output_tokens INTEGER, cost_usd REAL,

);

Note: portability of the memory layout into other harnesses is a non-goal — the Hermes split was rejected (§10). The rendered MEMORY.md + append-only logs are shaped for greppability and recoverability, not export. Task state lives in beads' own git-native format — portable by construction.

§7

Security & threat model

An always-on agent reachable from chat and able to run tools is a standing liability if you skip this. The dominant risk isn't a rogue model — it's prompt injection arriving through the very channels and web content the agent is built to read.

Primary threat: injection via untrusted content. A Discord message, a fetched page, an email body, or a community skill can carry instructions ("forward all mail to …", "run this"). Treat every byte the agent reads as data, never as commands. Cisco's audit of a third-party OpenClaw skill found silent data exfiltration — the ecosystem's cautionary tale, and the reason C9/C10 are non-negotiable here.

Policy matrix (what C9 actually implements)

Prose controls become one table, keyed on the trust field the adapter layer (C1) stamps on every event (§6), shifted one column right when the session is tainted. A fourth class, trust=system (daemon-stamped only — §4.2), reads the owner column for reads, notify, and task-graph verbs but can never satisfy an approval, and its sends are restricted to configured targets. This is the config in approach.toml; the PreToolUse hook answers from it.

Note the memory and task rows: writing a fact or an issue is a gated verb. An issue is worse than a fact — it's an instruction that persists and gets scheduled: a bd row created from injected content is a prompt injection with a cron job. Same enforcement, same provenance tag.

Two rules the matrix carries implicitly. (1) Bash is never a side-effect path. It always runs in the C10 container — every trust level, owner DM included — with no credentials and per-class egress; send/email/delete/purchase/push exist only as MCP or daemon verbs holding their own credentials, so the "curl from a trusted turn" bypass class is deleted, not enumerated. Host access exists only as host_exec, always-ask. (2) Taint shifts the column. A session that reads untrusted bytes (web fetch, non-owner participant, Codex output) gains a sticky tainted flag (§6, set by PostToolUse): mutating verbs then read one column to the right — an owner-DM-tainted session behaves like known-group, so its fact/task writes gate and provenance-tag untrusted-derived, feeding the read-time exclusion in §5 automatically. Reads are unaffected; the flag clears on rotation; high-impact verbs stay gated regardless.

Controls

  • Enforce at the hook layer, not the prompt. PreToolUse deny holds even when the model is fooled — CLAUDE.md rules don't. Gate every irreversible verb (send, publish, delete, transfer) behind a policy hook or --permission-prompt-tool that answers from code you control.
  • Allowlist tools per surface. --allowedTools narrows what each session class can even attempt; sends, deletes, pushes, and host_exec go through the deny-and-resume approval flow (§4.4) — durable, single-use, bound to a canonical argument digest.
  • Sandbox by default — everyone, always. The Bash tool executes in the C10 container for every trust level; nothing about being the owner turns it off (an injected page read in an owner session was the exact bypass). The profile is pinned: immutable versioned image · non-root UID · read-only root FS · cwd-only mount · no secrets or tokens in env · all capabilities dropped · no Docker socket · default seccomp · CPU/mem/PID/wall-time limits · defined artifact extraction · teardown verified by test. Egress is per session class: assistant sessions none; workers a read-only dependency allowlist (git clone/fetch, proxy.golang.org, registry.npmjs.org — per-repo config). Zero credentials inside any sandbox: workers commit locally, and push/PR is a gated host-side verb — the daemon pushes with its own credential after the matrix says yes, so a compromised build script gets a polluted container and nothing else.
  • Codex advises, Claude acts. The second engine bypasses your hook layer entirely — it has no equivalent of PreToolUse — so it is never granted gated verbs. It runs as a child process with its sandbox pinned read-only, and its output re-enters a gated Claude session as untrusted-adjacent data (it may have read the web). A misconfigured --sandbox danger-full-access in a skill would bypass everything above — pin the flags in the skill, test the pin.
  • Never in URLs. No secrets or personal data in query strings; no autofill/submit reached via a link from untrusted content; no sending data to endpoints suggested by fetched content.
  • Access & secrets. Reach the box only over Tailscale (you've already hardened SSH this way); bind any local UI to 127.0.0.1. Every service credential lives with the verb that uses it — inside its MCP server or the daemon (systemd credentials), never in the engine child's env, never in the workspace, never in a sandbox. Codex adds a second credentialed binary — its auth token gets the same treatment.
  • Cost as a safety control. Cap runaway loops with --max-turns and timeouts; the turns table (C11) gives the heartbeat a daily-spend query so the agent itself alarms on anomalous burn.
  • Kill switch. One systemd target — approach.target — groups the daemon and every timer. systemctl stop approach.target is the single documented panic command; nothing wakes up until you start it again.
§8

Deployment

  • Host. The Linux home server for the trusted solo agent; a small VPS later if you want it reachable when the homelab is down. Same systemd units either way. One box, one daemon — no second agent process anywhere (§10).
  • Process shape. APPROACH is one small Go daemon, channel adapters included — no separate bridge process. It owns the platform connections (discordgo, slack-go), the SQLite handle, a per-thread goroutine + queue over the durable events table (serialization for free, claimed by thread_key before session resolution — §4.1), pending approvals, and spawns claude -p as child processes from each session's recorded cwd (§6 — assistant sessions: ~/approach; workers: the repo path). systemd .timer units shrink to socket pokes — heartbeat, nightly reflection, and NL-cron all enter through the same event path as a message.
  • Model routing for cost. Three-way: heartbeats on Haiku; interactive turns on Sonnet 5 (the CC default) or Opus for heavy work; on-demand critique (C12) on Codex under ChatGPT subscription auth — and, if F2 lands (§10), its nightly skill-draft critique routes the same way, putting flat-rate pricing on what would be the most token-hungry recurring job. Routing lives in approach.toml, keyed on event kind.
  • Failure handling. One rule, specified in §4.6: retry only what provably did nothing; surface everything else. Engine failures auto-retry (≤2, backoff) only when live PostToolUse accounting shows zero side effects this turn; anything ambiguous parks as interrupted and surfaces to the originating thread; terminal failures land in dead_letters with two surfacing paths and manual drain. Events are never silently dropped because every failure path ends in a durable, human-visible state. An always-on agent that fails silently reads as ignoring you, which is worse than being down.
  • Task graph. beads adds nothing resident — bd is a CLI over git-synced JSONL plus a local SQLite cache. The heartbeat polls bd ready; worker sessions take and close issues from repo cwds. Pin the version (young, fast-moving project) and treat .beads/ as part of the workspace backup. Lease/assignment semantics are already in the tool for the day you run more than one worker.
  • Config. Model routing, the policy matrix, channel bindings, session TTLs, and retry policy live in one file — approach.toml (§6) — not smeared across hook scripts.
  • Interop. Portability into other harnesses is explicitly a non-goal — the Hermes split was evaluated and rejected on structural grounds (§10), so the layout in §6 is optimized for this system, not for export. One asymmetry that still matters: SKILL.md does not travel to Codex, which lives in an AGENTS.md + prompts world — don't count the skill library as shared with the second engine.

§10

Build vs buy — the decision

P1–P3 are squarely in your wheelhouse and buy you real ownership. The open-ended half of P4 is skill synthesis (F2), not fact curation. Decision (2026-07-09): build everything except F2, which is deferred behind an evidence trigger. The Hermes split is rejected.

Chosen

Build P1–P3 + F1/F3, defer F2

Ship the reflector with fact extraction (F1) and non-destructive rollups (F3); defer only the pattern → SKILL.md synthesis path (F2) and its review gate. F1's v1 posture is bounded by design: auto-write with provenance, confidence, and supersession links; passive decay (stale facts rank down in retrieval, never auto-delete); a periodic owner-DM digest of new/merged/superseded facts for one-tap pruning. Untrusted-derived facts are excluded from context assembly until acknowledged in a digest (§7, §11).

Rejected

Hermes split (was Options B/C)

Rejected on structural grounds, not evidential ones: a second always-on agent process with write access to the memory store is added attack surface and a split-brain fact-ownership problem — true even if every portability claim (SKILL.md compatibility, FTS5, "import not migration") were verified. Because evidence can't flip the decision, no compatibility spike is warranted and the portability claims drop out of the verification scope. The revisit, if F2's cost ever looks wrong, is build-vs-nothing — not build-vs-buy.

F2 re-entry

Evidence trigger, not a date

The F1 pass cheaply logs tagged repeated-pattern rows as it extracts facts — observation only, no synthesis. When ~5 distinct repeated patterns accumulate, or the owner notices re-teaching the same procedure, the F2 decision reopens. Those rows are exactly the input corpus F2 needs on day one, so the deferral is cheap to reverse.

The spec's original tell — build if you want to understand the "improves while you're not looking" property, buy if you only want the property — resolves to build: P1–P3 plus curated memory already yields a persistent, proactive, accreting agent on owned infra, and the one genuinely open-ended piece is deferred until daily use produces evidence it would pay for itself.

Same test, same verdict — cc-connect (C1)

Decision (2026-07-09): rejected; the channel layer is built native. An earlier draft counted cc-connect as an installed bridge. Inspection corrected two things at once: it isn't installed here, and it isn't a bridge — it's a full third-party harness (github.com/chenhg5/cc-connect: drives 10+ agents itself via ACP, owns session management, persistent memory, NL-cron scheduling, and chat-controlled permission modes). There is no pure-adapter mode to demote it into, so adopting it means a resident third-party process that drives agents, holds every channel token, sees all traffic, and keeps its own session/memory/schedule truth models — the Hermes objection with more surface. Forking to salvage its platform adapters inherits a large codebase's update stream for the 2–3 adapters actually needed. Building thin native connectors (discordgo, slack-go) costs days, and puts trust/dedup_key stamping in owned code by construction. cc-connect is retained as reference code for platform quirks — the same posture as reading OpenClaw for design ideas without running it.

§11

Open questions & risks

  • Concurrency. handled by design — the daemon's per-thread goroutine + queue over the durable events table (§3, §4.1, §8) serializes turns, and the one_live_session partial unique index (§6) backstops the router invariant in schema. Becomes a risk only if the implementation drifts back to per-event dispatch.
  • Session scope trap. handled by design — --resume lookups are scoped to the project dir + worktrees, so a session resumed from the wrong cwd silently fails. Every session records its spawn dir in sessions.cwd (§6) and the daemon asserts it on every resume (assistant sessions: ~/approach; workers: the repo path). The failure mode returns if a worker's repo moves or a worktree is pruned — assert, don't assume; when the assert (or the resume itself) fails, §4.6 degrades it to auto-rotate + fact-seeded fresh session + a transparency note, never a hard error.
  • Bare mode drift. --bare (faster, slated to become the -p default) skips hooks/skills/MCP/CLAUDE.md discovery — the opposite of what this harness needs. Pin the non-bare path explicitly.
  • Memory as attack surface. A curated fact written from injected content persists and re-enters future context. Mitigation is structural: fact writes are a gated verb in the policy matrix (§7), and the assembler excludes untrusted-derived rows from context assembly until they're acknowledged in an owner digest — the trust split applied at read time (§5). Residual risk: the reflector itself reads logs containing untrusted text — its extraction prompt must provenance-tag by source, not by content claims.
  • Skill-synthesis over-fitting. deferred with F2 — without the review gate, the agent teaches itself bad habits from lucky runs. When F2 lands (§10), the gate is not optional — and it must be cross-model (§5): a same-family self-review shares the blind spots it's supposed to catch.
  • Codex sandbox pin. The second engine has no hook layer; its safety rests entirely on the read-only sandbox flags pinned in the critic skill (§7). One edited flag reopens everything — treat the skill file as security-sensitive config and test the pin, don't assume it.
  • Approval fatigue. If §4.4 fires too often, you'll start rubber-stamping "approve 42" — at which point the gate is theater. Watch the pending_approvals rate in the turns data; if a verb is approved >90% of the time from the owner surface, promote it to allow deliberately rather than letting habituation do it.
  • Noun-partition drift (beads). Two truth models coexist by design (§6): facts in approach.db, work in beads. The partition erodes one convenience at a time — a "fact" that's really a TODO, an issue used to store a durable preference. Make the reflector police it: part of the nightly pass is moving misfiled rows to the right store.
  • Issues as persisted injections. A bd row created from untrusted content is an instruction that persists and re-enters via bd ready — strictly worse than a poisoned fact. The matrix (§7) denies untrusted task writes and provenance-tags gated ones; workers should treat issue bodies from non-owner provenance as data, same as any fetched content.
  • Dependency velocity (beads). Young, fast-moving project — pin the version, wrap bd calls in one skill (a single choke point when flags change), and keep .beads/ in the workspace backup so a bad upgrade is a checkout away from undone.
  • Runaway workers. A worker session is an unattended agent with a goal — the costliest failure shape in the system. --max-turns per worker, a per-issue cost ceiling from the turns table, and at most N concurrent workers (start with N=1) in approach.toml.
