§1

Purpose & scope

Reconstruct the useful properties of OpenClaw / Hermes on infrastructure you own and understand, so the "always-on, self-improving assistant" behavior isn't rented from a framework whose harness you can't see into.

Goals

  • Persistence. Two deliberate shapes, not one. Conversational threads: a message (or a webhook targeting a thread) continues the same live session hours or days later — not a fresh stateless invocation. Scheduled runs: heartbeats and cron fires start deliberately fresh sessions over durable state (§4.2) — continuity lives in facts, schedules, and the task graph, not in an ever-growing timer session.
  • Proactivity. The agent wakes itself on a schedule, decides whether anything is worth surfacing, and stays quiet when it isn't.
  • Accretion. Facts and successful patterns persist across sessions and are pulled back in on relevance — the agent's competence compounds.
  • Ownership. Everything runs as plain files, SQLite, and systemd units on your box, behind Tailscale, with no vendor control plane.
  • Orchestration. Multi-session work — including coding-agent work across repos — lives in a git-native task graph (beads). Ready-work detection decides what gets picked up; blocked work parks with its blocker instead of rotting in prompts or leaking into memory.

Non-goals

  • Matching OpenClaw's 20+ hardened channel adapters. You wire the two or three surfaces you actually use.
  • A polished install wizard or GUI. This is a homelab system for one operator.
  • Multi-tenant / SaaS shape. That's a separate decision (the BYOK angle you've mulled), explicitly out of scope here.
§2

The gap, precisely

Claude Code + skills already covers the expensive-to-build part: the reasoning engine and its tool loop. The delta is the harness around it — channels, persistence, scheduling, and a curated memory loop.

What the engine already gives you (P0 — installed)

  • Headless agent loop. claude -p "…" runs the full tool-using loop and exits; --output-format stream-json gives you an event stream the daemon relays out to channels.
  • Session resume. Pin a UUID at spawn with --session-id, then --resume <id> on every later turn (capture from --output-format json also exists, but the harness pre-pins — §4.1). This is the primitive the whole persistence layer is built on — you don't reimplement conversation state, you index into Claude Code's.
  • Deterministic hooks. A 30-event lifecycle (verified 2026-07-10: SessionStart, UserPromptSubmit, PreToolUse, PermissionRequest, PostToolUse, Stop, SessionEnd, PreCompact …) that fires regardless of model cooperation — count is version-sensitive, so still pin the CLI and treat the enrolled hook set as config. This is your enforcement + reflection substrate.
  • MCP + skills. .mcp.json servers and .claude/skills/*/SKILL.md — plain-file skills that live in the workspace and travel with git.
  • Session transcripts as durable state. Conversation history persists under ~/.claude/projects/<proj>/ as --resume-able transcripts — the primitive the persistence layer indexes into. Note the curated memory store (§5–§6) is built fresh on SQLite, not extended from Claude Code's own MEMORY.md auto-memory (whose behavior is an unverified assumption, §11) — APPROACH owns its facts.

What's missing (the build)

  • Channel adapters — thin, native connectors (discordgo, slack-go) for the 2–3 surfaces you actually use, normalizing every entry point into the §6 event contract and stamping trust/thread_key/dedup_key in code you own. Small per surface. (cc-connect was evaluated for this role and rejected — §10.)
  • Session router + scheduler — mapping channel threads to sessions, and a heartbeat that invokes the agent unprompted. Thin-to-small; mostly plumbing over primitives that already exist.
  • Curated memory + recall — agent-authored facts, relevance retrieval (FTS5), and log compaction that you control rather than accept as a black box. Moderate.
  • Self-improvement loop — reflection that extracts durable facts and synthesizes new skills from successful runs. The genuinely hard part is the synthesis half — deliberately deferred (§10): v1 ships fact extraction and rollups; skill synthesis waits on an evidence trigger.
  • Policy + sandbox — because inbound messages and fetched web content are untrusted, this is not optional. Config-heavy but must-do.
Read this as a build map with one decided fork. Sections §3–§8 spec the whole system; §9 sequences it; §10 records the build-vs-buy decision: build it, defer skill synthesis (F2) behind an evidence trigger — the Hermes split was evaluated and rejected on structural grounds.

Shape decision. APPROACH is a single small Go daemon — channel adapters included — not a per-event dispatcher. Four requirements force it: per-thread serialization (a lock that must span processes), live stream-json relay, durable pending approvals, and a durable event queue written on receipt (§4.1). One resident process with a per-thread goroutine + queue over the events table makes all four trivial; systemd timers shrink to socket pokes, and the daemon spawns each session from its recorded cwd (§6) — which also closes the --resume cwd-scope trap (§11).
§4

Control flows

4.1  Inbound message

The hot path. First act on receipt: persist the event row (§6) — gateway channels don't redeliver, so receipt is the last moment durability is free; everything after is recoverable from the table. The engine never invents a session — the router resolves it from the store — and the Stop hook records the turn asynchronously so the reply isn't blocked on memory writes. Queues are keyed by thread_key, not session_id, and claimed before resolution: two simultaneous first-messages to a new thread serialize in the queue, and the sessions schema's partial unique index (§6) backstops the one-active-session-per-thread invariant even against daemon bugs. In-memory queues are just an index over unprocessed event rows — per-thread FIFO by receipt order, rebuilt from the table on restart. The reply leg is at-least-once and durable: the rendered reply is written to a deliveries outbox row (§6) before the first send attempt, the event advances to replied only after the adapter confirms the platform accepted it, and unacked delivery rows re-send on restart — a crash after compose-but-before-ack re-sends from the persisted payload, so a rare duplicated chat message is accepted; a silently eaten reply is not. Every outbound message uses this outbox — proactive heartbeat/cron notifies (§4.2) included, not just replies. And every side-effecting tool call is journalled: the PreToolUse gate writes a tool_attempts row in state started before the call runs and flips it to done/failed after (§6), so recovery reasons from what provably started, not from aggregate counts a crash can lose.

4.2  Heartbeat (proactivity)

A timer, not a message, drives this. The agent reads a checklist and exercises judgment on a schedule; a no-op result is swallowed so silence is the default. Run it on a cheap model (Haiku) to keep idle cost near zero. Three protocol decisions: heartbeats run in a fresh session every fire (a persistent heartbeat session accumulates cruft and cost); "something to say" is expressed as a notify tool call, never a sentinel string parsed out of free text — no tool call means silence; and the prompt reads/writes a notifications ledger (item_key, target, notified_at, snooze_until — keyed per recipient, §6) so a 30-minute timer doesn't re-nag the same deadline 16 times a day. NL-cron ("brief me at 8am") is a schedules row the agent inserts via a skill — the scheduler fires rows, not just HEARTBEAT.md. Every fire has a manufactured identity — (schedule_id, occurrence_time) is the event's idempotency key — and misfires coalesce with a staleness flag: after downtime, at most one catch-up event fires per schedule, keyed to the newest missed occurrence and carrying missed_count + lateness so the model judges whether a 3-hour-late brief still ships; older occurrences are recorded as skipped, never enqueued. Three more contract points. Trust: system-generated events (heartbeat, cron, reflection triggers) carry trust=system — stampable only by the daemon itself, never by an adapter; it reads like owner for reads, notify, and task-graph operations, but can never satisfy an approval, and its sends are restricted to configured targets. No escalation via schedules: a cron fire runs at min(system, schedules.created_trust) (§6, §7) — a schedule inserted from a known or gated request fires at that trust, never system, so lower-trust content can't launder itself into a privileged recurring job. Creating or editing a schedule is itself a gated matrix verb (§7). Time: each schedule stores an IANA timezone; cron expressions are DST-aware local wall-clock (a nonexistent 2:30am fires at the next valid instant; a repeated one fires once), and occurrences are recorded UTC + zone. Overlap: max_overlap = 1 by default — an occurrence due while the previous fire's session still runs is recorded as a skipped event row, never enqueued; fires and skips alike are event rows, so the per-fire audit trail is the events table itself, and a chronically-skipping schedule shows up in the digest as a tuning signal. One standing checklist item: bd ready — unblocked task-graph work is surfaced (or picked up, §4.5) through the same judgment-on-a-schedule loop. The heartbeat is the only component that asks beads what's ready; beads never initiates.

4.3  Context assembly

What actually goes into each turn. Keep it lean: the model gets a skills list (names + descriptions, not bodies) and only the facts that retrieve against the current input — this is where a black-box harness quietly bloats your token bill. Mechanism is pinned: the UserPromptSubmit hook, not a prompt prefix. And because --resume already carries the session's history, the full fact package is injected only on session start/rotation; per-turn injection is the retrieval delta — facts matching this input that aren't already in-session. "Already in-session" is a ledger, not a guess: the assembler records every injected fact id in session_facts (§6), and each turn injects FTS hits minus that set — ranked by match score, confidence, and recency, capped by a top-k / token budget in approach.toml (start k=8). Retrieval is trust-filtered by the thread floor, in both directions. The filter keys off sessions.trust_floor — the least-trusted participant the thread can admit (§6), not the current sender — because owner and untrusted senders in a group share one --resume transcript, so a fact injected on the owner's turn is readable on a later untrusted turn. The query excludes (a) rows where trust_floor < fact.min_trust — owner-private facts (default min_trust=owner) therefore never assemble into any multi-party session, only into a single-owner DM whose floor is owner; (b) unacked untrusted-derived rows (§5); and (c) superseded rows (superseded_by IS NOT NULL) — ranked-down is not enough, a stale contradictory fact must not re-enter at all. Raw log_entries retrieval is filtered the same way (log rows carry their own min_trust, §6). system reads as owner. Re-paying the whole fact block every turn is exactly the bloat this figure warns about.

4.4  Approval round-trip (gated verbs)

The hardest flow in the system, so it's pinned: deny-and-resume. Hooks are synchronous with timeouts — the gate never holds a turn open waiting for a human. It records the intended call, denies with a reason, and lets the turn end gracefully; your approval arrives later as just another inbound event that resumes the session. Approvals are durable rows, so they survive restarts, and a gated call can never fire without a matching approved row. The binding is cryptographic, not textual: the row stores the canonical tool name and a digest of the normalized JSON arguments (the rendered string is display-only), plus the requesting event id, session, and timestamps; the approval id doubles as the nonce. Authorization: an approval event is valid iff its sender's owner_id resolves to the row's stored authorized_owner_id (§6) and arrives on a strong-auth channel (§7 — Discord/Slack/Tailscale UI; never spoofable SMS/email, which can't stamp owner trust or approve at all). Any strong owner surface works, not just the requesting one (approve from Slack what Discord asked; it's the same canonical principal). The approval is enqueued onto the bound session's thread queue (from pending_approvals.session_id), not the arriving surface's — so the serialization the re-check relies on holds even across surfaces. For push/PR verbs the digest also binds the resolved commit_range and the request renders a diffstat, so the owner approves specific commits, not a blind git push; a push whose range resolves differently starts a fresh cycle. Expiry: 1 hour — a stale "approve 42" can't fire tomorrow against forgotten context; expired means a fresh request cycle. Rotation voids approvals: if the bound session rotates (idle TTL, turn cap, resume-failure) before the approval arrives, the outstanding approval is cancelled (status=expired) and the action re-requests on the successor session — a stale approval never fires into a fresh, unexpected session. Match-and-consume is one transaction that re-evaluates the current matrix — including the session's tainted flag — at execution time, not just the digest; turn serialization guarantees no PostToolUse (hence no taint change) can slip between the deny that ended the turn and the approve that resumes it, so the re-check is belt-and-suspenders, not a race. If the resumed model re-issues a call whose digest differs from what was approved, that is a new approval cycle, never a fuzzy match.

4.5  Task orchestration (coding-agent work)

Multi-session work lives in beads (C13), not in prompts. A ready issue becomes a worker session: a row in the same sessions table, keyed task:<issue-id>, spawned from the repo's cwd (stored per session — §6; this is what forces the sessions.cwd column, since --resume is scoped to the spawn directory). Workers run under the same policy matrix and the same per-thread queue; a worker surviving a reboot is just C4 persistence applied to work instead of conversation. Done → bd close -r with a closing reason the reflector (§5) can read; stuck → bd dep add <issue> <blocker> (or bd update --defer), and the issue drops out of ready until its blocker clears — parked, not nagging, not leaking into facts.

The claim is atomic because the daemon is the lock, and bd runs daemon-side. Workers never invoke bd from inside their sandbox — the sandbox has zero credentials and can't reach the Dolt remote (§7). Instead, bd mutations against the single central store (~/approach/.beads, §6) are a gated daemon-side verb (the "Task graph writes" matrix row, §7), executed with the daemon's credential. One serialized claim section: check max_workers (start N=1) → insert the task:<id> session row (two schema indexes bite here — one_live_session stops a double-spawn, one_worker_per_repo stops two workers sharing a repo cwd) → bd update --claim as the durable record. Two heartbeat fires can both see an issue as ready; only one exits the claim section with it. bd's claim field is a record, not a mutex (verified: atomic field write, no lease/expiry — §8), so a crashed worker's claim is released by the restart sweep below, not by a timeout. The session row (approach.db) and the bd claim (Dolt) are two stores, so the write is a reconciled sequence, not a single transaction: the daemon writes the session row first (the schema-enforced lock), then the bd claim; if it crashes between, the restart sweep is the compensating action — a task:* session with no matching bd claim gets the claim written (or the session released), so the two stores converge. The bd row also carries the originating thread as a provenance label, so a worker's work is always traceable to its request. Each worker session records its origin thread. A worker turn is a distinct event kind=task carrying trust=system-worker — it routes to the interactive model tier, never Haiku (a cheap model on unattended coding work is the §11 runaway risk made worse), reads as system for bd/repo operations, and can never satisfy an approval. Restart reconciliation: on boot, sweep active task:* sessions — if the transcript resumes (§4.6 assert), the worker re-enters its queue with a "you were interrupted; reassess and continue" nudge, half-done work visible via git status in its own repo; if the transcript is gone, mark resume_failed, release the bd claim (the issue returns to ready), and post a note to the origin thread. Mid-task events follow §4.6 — no side-effect replay, ever.

4.6  Failure & recovery

One rule unifies every failure path: retry only what provably did nothing; surface everything else. Events are never silently dropped because every path below ends in a durable state a human can see.

  • Daemon crash mid-turn. On restart, events found in processing become interrupted — never auto-rerun. The side effects of a half-finished turn are mechanically unknowable (the email may or may not have gone out), so the daemon posts to the originating thread: what it was doing, and an offer to retry. Interrupted events park out-of-band — the thread's queue keeps flowing; a "retry" re-enqueues the old event at the tail, executing against current session state (--resume re-reads the transcript, which is exactly what absorbs the staleness). A frozen assistant that won't answer "hello?" until you adjudicate a crash report would be the worse failure.
  • Engine failure, daemon alive (non-zero exit, timeout, rate limit). The daemon reasons from the tool_attempts journal (§6), not aggregate counts: if the turn has no tool_attempts row for a side-effecting verb, auto-retry (up to 2, backoff, honoring rate limits). If any side-effecting attempt exists — even one still in started (a crash could have lost its PostToolUse) — it is ambiguous: no retry unless the verb carried an idempotency_key that makes a repeat provably safe; otherwise interrupted + surface. Timeouts kill the child, then apply the same test.
  • Reply undelivered. Turn completed but the platform send was never confirmed (§4.1): re-send on restart. At-least-once, duplicate accepted.
  • Resume failure. The pinned session UUID points into Claude Code's store under ~/.claude/projects/; if that transcript is gone or unreachable (GC, corruption, moved cwd — §11), the row is marked resume_failed (a rotation cause, kept for forensics), a fresh session is created and seeded with the assembler's full fact package, the event processes there, and the reply carries a one-line note: "lost this thread's conversation history — facts intact, starting fresh." Transcript loss degrades to amnesia-with-notes, not an error; facts in approach.db are the durable memory by design. (Backing up the Claude-side store is a §6 ownership question.)
  • Dead letters. Terminal failures land in dead_letters (§6): retry budget exhausted, malformed or unroutable events, and interrupted events whose surface attempt itself failed. Exit is two independent paths — one owner-DM notification on entry (through the normal outbound leg), plus a standing heartbeat checklist item for unacknowledged rows (via the notifications ledger, so no re-nag). Draining is deliberately manual: a requeue/discard command per row. "Dead" means the machine has given up and a human decides.
§5

Memory & the learning loop

The hardest slice of the build — the part that separates "Claude Code with a cron job" from something that genuinely improves. Worth being honest about why it's hard.

Why it's the hard part

  • Fact curation is lossy and adversarial to itself. Naive "append what you learned" produces contradictory, duplicated, stale facts within days. You need dedup/merge, supersession ("X moved from A to B"), and decay — a small ontology problem, not a prompt. The facts table in §6 makes these row operations instead of markdown surgery, which is why SQLite (not MEMORY.md) holds the truth. V1 posture: the reflector auto-writes facts with provenance, confidence, and supersession links; decay is passive (stale facts rank down in retrieval, never auto-delete); a periodic owner-DM digest lists new/merged/superseded facts for one-tap pruning — human oversight without a blocking queue. Facts derived from trajectories containing untrusted content are stored immediately but excluded from context assembly until acknowledged in a digest (the acked column, §6) — the trust split applied at read time (§7, §11). Crucially, provenance is computed by the daemon, not the model: an extracted fact's provenance is the max-untrust over the log_entries the reflection job spanned — if any source row in the watermark range was untrusted, the fact is untrusted-derived regardless of what the reflector's output claims. The one place the defense could rest on model cooperation is closed mechanically.
  • Skill synthesis needs a taste filter — which is why it's deferred. Auto-generating a SKILL.md from one lucky run yields brittle, over-fit skills. The honest version watches for repeated success, drafts, and puts a review gate in front of anything that lands in the skill dir — an open-ended tuning project, deferred behind the evidence trigger in §10. The gate's cheap concrete mechanism, when it lands, is cross-model critique: Claude drafts the skill, codex exec (C12) attacks it with a different model lineage — de-correlated errors are exactly what a same-family self-review can't give you — and you approve. C12 ships at launch as an on-demand critic, so the mechanism is proven before F2 needs it.
  • Retrieval quality caps everything. Start with SQLite FTS5 — fast, transparent, no embedding infra. Add vectors only if lexical recall provably misses. Resist the urge to over-build here first.
  • Compaction is where memory rots silently. Summarizing yesterday's logs discards detail you can't get back. log_entries rows are append-only; compaction writes additional rollup rows (rendered into rollups/), never destructively — the raw rows survive every summary pass.
  • Reflection is a job, not a hook side effect. Every run is a reflection_jobs row (§6) under a single-holder lease with a durable watermark over log_entries + closed sessions. The nightly sweep is the correctness guarantee — it reflects everything from the last watermark to now, so nothing is ever missed; Stop/SessionEnd triggers are opportunistic freshness enqueues of the same job kinds, deduped by the unique (kind, range) key — nice when they run, harmless when the daemon dies (tonight's sweep covers it). A fact merge is one transaction: upsert the merged fact by its deterministic fact_key (§6, so a crash-retried extraction never duplicates), set superseded_by on the olds, copy fact_sources rows (unique per source), advance the per-kind checkpoint — all-or-nothing, no partially-merged state visible. One residual the transaction can't reach: a superseded fact already injected into a live session's transcript stays in that --resume history until rotation. So on a supersession that hits an active session_facts row, the assembler injects a one-line correction on the next turn (and forces rotation for a high-confidence contradiction) rather than letting the stale version stand.
Decision hinge — resolved. C4/C5/C6 are bounded plumbing (S–M) over primitives you already have. Within C8, the open-ended part is F2 skill synthesis; F1 fact curation is bounded by the v1 posture above (auto-write + digest + append-only means curation errors are recoverable, not compounding). §10 records the decision: build F1/F3 now, defer F2 behind an evidence trigger, Hermes rejected.
§6

Data model & layout

Everything is plain files + one SQLite db — greppable, git-diffable, recoverable.

Source-of-truth rules. Every piece of structured state has exactly one owner — facts living in both a markdown file and an index is the classic setup for silent divergence. So: SQLite is truth; MEMORY.md is a rendered view, regenerated after each reflection pass and committed to git. Dedup, supersession, decay, and provenance are row operations; the greppable/diffable goal survives because you diff the render. A second truth model exists on purpose — beads' embedded Dolt db (synced via refs/dolt/data; JSONL is a passive export) — kept honest by a strict noun partition: facts = knowledge (approach.db), beads = work. If a row could be phrased as "do X," it's an issue, not a fact; if it's true after the work is done, it's a fact, not an issue.

Identity & trust resolution (the root of §7)

Every gate and the approval flow reduce to one question: is this sender the owner, someone known, or untrusted? That is not inferred — it is a table lookup. A hand-enrolled identities table maps each (channel, native_id) to a trust level and, for owner rows, to a single canonical owner_id. The adapter looks up every inbound sender; a hit stamps the row's trust, a miss stamps untrusted — deny-by-default, so an unknown Discord DM is untrusted, not owner. Because all of the owner's surfaces share one owner_id, "approve from Slack what Discord asked" (§4.4) is an owner_id match, not a channel match. You enroll your own handful of ids once; a homelab has one owner and a few known people. Channel auth is graded, because identity is only as strong as the channel that carries it. Discord/Slack ids are platform-authenticated and Tailscale-fronted surfaces are network-authenticated (auth=strong); an SMS e164 or email From is trivially spoofable (auth=weak). A weak channel's stamped trust is clamped to at most known, read-only, and it can never satisfy an approval — so a spoofed SMS "from" your number cannot resolve to owner or rubber-stamp "approve 42." Session trust_floor: the adapter's per-event trust is one input; a session also carries a trust_floor (§6) = the least-trusted party the thread can admit, and confidentiality (§4.3) keys off the floor so a private fact never lands in a transcript a lower-trust participant will later read via --resume.

CREATE TABLE identities (            -- hand-enrolled; seeded from approach.toml
  channel   TEXT NOT NULL,           -- discord | slack | sms | …
  native_id TEXT NOT NULL,           -- discord user_id, slack user, e164 …
  trust     TEXT NOT NULL,           -- owner | known  (no row ⇒ untrusted, deny-by-default) —
                                     --   CLAMPED by the channel's auth class below
  owner_id  TEXT,                    -- canonical principal: same value across ALL of the owner's
                                     --   surfaces — unifies cross-surface approval (§4.4)
  label     TEXT,
  PRIMARY KEY (channel, native_id)
);
-- Channel auth grading (§7): a channel's sender identity is only as trustworthy as its auth.
-- strong = platform-authenticated (discord, slack) or network-authenticated (tailscale UI);
-- weak = spoofable (sms e164, email From). A weak channel is CLAMPED to at most 'known',
-- read-only, and can NEVER stamp owner trust or satisfy an approval (§4.4). Config in approach.toml:
--   [channels.sms]   auth = "weak"     ⇒ max_trust = known, may_approve = false
--   [channels.discord] auth = "strong" ⇒ max_trust = owner, may_approve = true

Durability, backup, recovery

  • Two durability tiers, named honestly. Tier-1 (loss = broken): approach.db, .beads/, and the rendered workspace — the facts/work/config truth. Tier-2 (loss = degraded): the Claude-side transcript store under ~/.claude/projects — a store APPROACH indexes into but does not own (Claude Code controls its format and GC). It's included in backups because it's small and cheap, but restore is best-effort: §4.6's amnesia-with-notes is the contract when a transcript is gone. This is not hypothetical — Claude Code silently garbage-collects transcripts after cleanupPeriodDays (default 30) with no warning or recovery (verified 2026-07-09); raise it in settings.json and let the heartbeat orphan sweep flag sessions rows whose transcript has vanished, so GC drift is seen before you hit it mid-conversation.
  • SQLite settings. WAL mode, synchronous=NORMAL, busy_timeout set — single box, single writer; the snapshot below is the durability backstop. state/ is 0700, the db 0600.
  • Migrations. Numbered SQL migrations embedded in the daemon binary, applied in one transaction at startup; the daemon refuses to start on a schema version newer than it knows — downgrade protection beats mystery corruption.
  • Backup mechanism. Nightly, after reflection: VACUUM INTO a dated snapshot (SQLite's own consistent-copy path — safe against a live WAL), then restic ships ~/approach + snapshots + ~/.claude/projects offsite (NAS/B2); retention 7 daily + 4 weekly. RPO for approach.db = the snapshot interval (24h as specified — tighten by running the snapshot more often, or add Litestream WAL-streaming if a lost day ever bites). Beads is separate: its RPO is the bd dolt push cadence, which the daemon runs after every gated task-graph write, so committed issue state is at-remote within a turn — but that is a push cadence, not "continuous replication," and a push that hasn't run yet is only as safe as the nightly snapshot.
  • Restore is a drill, not a document. Order: workspace checkout → newest db snapshot → transcripts → bd doctor. A scripted approach restore-drill restores into a temp dir and asserts sessions/facts/events row counts — run quarterly. An untested backup is a hope, not a control.

Security & threat model

An always-on agent reachable from chat and able to run tools is a standing liability if you skip this. The dominant risk isn't a rogue model — it's prompt injection arriving through the very channels and web content the agent is built to read.

Primary threat: injection via untrusted content. A Discord message, a fetched page, an email body, or a community skill can carry instructions ("forward all mail to …", "run this"). Treat every byte the agent reads as data, never as commands. Cisco AI Defense's analysis (2026-01-28) of a top-ranked third-party OpenClaw skill found silent data exfiltration — a hidden curl to an attacker server, output to /dev/null — plus prompt injection, in a skill downloaded thousands of times. The ecosystem's cautionary tale, and the reason C9/C10 are non-negotiable here.

Policy matrix (what C9 actually implements)

Prose controls become one table, keyed on the session's effective trust = min(the sender's identities-stamped trust clamped by channel auth, the session's trust_floor, and any taint step) over the lattice owner>known>untrusted (§6, §4.3). Two daemon-only classes sit alongside the three adapter-stampable ones: system (heartbeat/cron/reflection — §4.2) and system-worker (task turns — §4.5) both read the owner column for reads, notify, and task-graph verbs but can never satisfy an approval, and system sends are restricted to configured targets. Schedule writes are themselves a gated row (so untrusted content can't plant a recurring job), and a fire's trust is capped at its creator's (§4.2). This is the config in approach.toml; the PreToolUse hook answers from it.

Trust is a lattice; four rules the matrix carries implicitly. Trust is ordered owner > known > untrusted (with daemon-only system reading as owner); "one column right" and "min/floor" are operations on that lattice, and a session's effective trust is the minimum of its trust_floor and any taint shift. (1) Bash is never a side-effect path. The Bash tool is wired — via .claude/settings.json permission config, the same hook layer that gates everything else — to execute only in the C10 container, every trust level including owner DM, no credentials, per-class egress; send/email/delete/purchase/push exist only as MCP or daemon verbs holding their own credentials, so the "curl from a trusted turn" bypass class is deleted. Host access exists only as host_exec, always-ask. (2) Reads and writes are scoped, not free. Host-side Read/Grep and Edit/Write/NotebookEdit are confined to the session's cwd subtree with a hard path denylist — ~/.ssh, credentials, and the harness's own control surface (.claude/ hooks·settings·skills, .codex/, approach.toml) are refused for both read and write — so neither credential exfiltration nor the worst escalation, the agent editing its own PreToolUse hook to disable the gate, is expressible in-band. (3) Taint follows content, not the tool. A session gains the sticky tainted flag (§6) the moment it ingests externally-authored content — an inbound non-owner prompt or attachment (stamped at receipt, pre-execution), a tool result from an untrusted-returning MCP server (email body, calendar invite, Drive doc — enumerated in approach.toml, default untrusted), a web fetch, or a Codex web read. Keying on the tool alone would miss the email vector the threat model names, so it keys on the data. Mutating verbs then drop one lattice step; fact/task writes provenance-tag untrusted-derived, feeding the §5 read-time exclusion. A read-only Codex critic call does not itself taint, so draft→critique→approve→act survives. (4) Web fetch egress is hook-enforced. Once a session is tainted, outbound fetch URLs are query-stripped or held to a host allowlist at the hook layer — an injected https://attacker/?d=<secret> can't exfiltrate on the model's say-so. The flag clears on rotation; high-impact verbs stay gated regardless.

Controls

  • Enforce at the hook layer, not the prompt. PreToolUse deny holds even when the model is fooled — CLAUDE.md rules don't. Gate every irreversible verb (send, publish, delete, transfer) behind a policy hook or --permission-prompt-tool that answers from code you control. (--permission-prompt-tool is documented — verified 2026-07-10; note as of v2.1.199 it cannot approve an MCP tool marked as requiring user interaction, an allow there converts to deny, so still test the flag against your pinned version in the C9 drills.)
  • Allowlist tools per surface. --allowedTools narrows what each session class can even attempt; sends, deletes, pushes, and host_exec go through the deny-and-resume approval flow (§4.4) — durable, single-use, bound to a canonical argument digest.
  • Sandbox by default — everyone, always. The Bash tool executes in the C10 container for every trust level; nothing about being the owner turns it off (an injected page read in an owner session was the exact bypass). The profile is pinned: immutable versioned image · non-root UID · read-only root FS · cwd-only mount · no secrets or tokens in env · all capabilities dropped · no Docker socket · default seccomp · CPU/mem/PID/wall-time limits · defined artifact extraction · teardown verified by test. Egress is per session class: assistant sessions none; workers a read-only dependency allowlist (git clone/fetch, proxy.golang.org, registry.npmjs.org — per-repo config). Zero credentials inside any sandbox: workers commit locally, and push/PR is a gated host-side verb — the daemon pushes with its own credential after the matrix says yes, so a compromised build script gets a polluted container and nothing else.
  • Codex advises, Claude acts. Codex CLI now has its own hook lifecycle (verified 0.144.0: PreToolUse deny-gating, UserPromptSubmit context injection) — but the rule stands on a different leg: one enforced policy stack, not two. Maintaining a second, parallel hook/policy configuration for a second engine doubles the surface where one edited line reopens everything, so Codex is simply never granted gated verbs. It runs as a child process with its sandbox pinned read-only, and its output re-enters a gated Claude session as untrusted-adjacent data (it may have read the web). A misconfigured --sandbox danger-full-access in a skill would bypass everything above — pin the flags in the skill, test the pin.
  • Egress is enforced, not requested. The prior "don't put secrets in URLs" guidance is now the hook-enforced fetch-egress control above (rule 4): a tainted session's outbound URLs are query-stripped or allowlisted at the hook, so exfiltration doesn't depend on the model declining to do it.
  • Access & secrets. Reach the box only over Tailscale (you've already hardened SSH this way); bind any local UI to 127.0.0.1. Every service credential lives with the verb that uses it — inside its MCP server or the daemon (systemd credentials), never in the engine child's env, never in the workspace, never in a sandbox. Codex adds a second credentialed binary — its auth token gets the same treatment.
  • Cost as a safety control. Cap runaway loops with --max-turns and timeouts; the turns table (C11) gives the heartbeat a daily-spend query so the agent itself alarms on anomalous burn.
  • Kill switch. One systemd target — approach.target — groups the daemon and every timer. systemctl --user disable --now approach.target is the single documented panic command; nothing wakes up — including across logins and reboots — until you enable --now it again. (A plain stop halts everything too, but the target is WantedBy=default.target and would re-activate with the next session; see deploy/systemd/README.md.)

Deployment

  • Host. The Linux home server for the trusted solo agent; a small VPS later if you want it reachable when the homelab is down. Same systemd units either way. One box, one daemon — no second agent process anywhere (§10).
  • Process shape. APPROACH is one small Go daemon, channel adapters included — no separate bridge process. It owns the platform connections (discordgo, slack-go), the SQLite handle, a per-thread goroutine + queue over the durable events table (serialization for free, claimed by thread_key before session resolution — §4.1), pending approvals, and spawns claude -p as child processes from each session's recorded cwd (§6 — assistant sessions: ~/approach; workers: the repo path). systemd .timer units shrink to socket pokes — heartbeat, nightly reflection, and NL-cron all enter through the same event path as a message.
  • Model routing for cost. Three-way: heartbeats on Haiku 4.5; interactive turns on Sonnet 5 (pinned in approach.toml — the CLI's default is settings-derived, don't rely on it) or Opus for heavy work; on-demand critique (C12) on Codex under ChatGPT subscription auth. Assumption, marked: subscription Codex is bundled-but-metered (rolling agentic usage limits; token-based accounting since Apr 2026), and no ToS clause explicitly blesses recurring unattended jobs — so if F2's nightly critique lands, treat subscription routing as best-effort with API-key billing as the fallback, not as guaranteed flat-rate. Routing lives in approach.toml, keyed on event kind.
  • Failure handling. One rule, specified in §4.6: retry only what provably did nothing; surface everything else. Engine failures auto-retry (≤2, backoff) only when live PostToolUse accounting shows zero side effects this turn; anything ambiguous parks as interrupted and surfaces to the originating thread; terminal failures land in dead_letters with two surfacing paths and manual drain. Events are never silently dropped because every failure path ends in a durable, human-visible state. An always-on agent that fails silently reads as ignoring you, which is worse than being down.
  • Task graph. beads adds nothing resident — bd is a CLI over an embedded Dolt db (truth; syncs via refs/dolt/data; JSONL is a passive export). The heartbeat polls bd ready; worker sessions take and close issues from repo cwds. Pin the version (young, fast-moving project) and treat .beads/ as part of the workspace backup. Claims are atomic field writes, not leases (verified v1.1.0): bd update --claim is atomic and idempotent, but there is no lease renewal or expiry — a crashed worker's claim persists until changed, which is exactly why §4.5's restart reconciliation releases claims and bd stale is a heartbeat checklist item.
  • Config. Model routing, the policy matrix, channel bindings, session TTLs, and retry policy live in one file — approach.toml (§6) — not smeared across hook scripts.
  • Backup. Nightly snapshot + restic offsite, two durability tiers, quarterly restore drill — specified in §6 (Durability, backup, recovery). The panic command's quiet sibling: approach restore-drill.
  • Interop. Portability into other harnesses is explicitly a non-goal — the Hermes split was evaluated and rejected on structural grounds (§10), so the layout in §6 is optimized for this system, not for export. One earlier asymmetry has closed (verified 2026-07-09): Codex now supports SKILL.md skills via the shared agent-skills standard (.agents/skills), alongside AGENTS.md — so the skill library is largely shareable with the second engine, though Codex-bound skills get no hook enforcement from this harness and stay read-only advisory (§7).

§10

Build vs buy — the decision

P1–P3 are squarely in your wheelhouse and buy you real ownership. The open-ended half of P4 is skill synthesis (F2), not fact curation. Decision (2026-07-09): build everything except F2, which is deferred behind an evidence trigger. The Hermes split is rejected.

Chosen

Build P1–P3 + F1/F3, defer F2

Ship the reflector with fact extraction (F1) and non-destructive rollups (F3); defer only the pattern → SKILL.md synthesis path (F2) and its review gate. F1's v1 posture is bounded by design: auto-write with provenance, confidence, and supersession links; passive decay (stale facts rank down in retrieval, never auto-delete); a periodic owner-DM digest of new/merged/superseded facts for one-tap pruning. Untrusted-derived facts are excluded from context assembly until acknowledged in a digest (§7, §11).

Rejected

Hermes split (was Options B/C)

Rejected on structural grounds, not evidential ones: a second always-on agent process with write access to the memory store is added attack surface and a split-brain fact-ownership problem — true even if every portability claim (SKILL.md compatibility, FTS5, "import not migration") were verified. Because evidence can't flip the decision, no compatibility spike is warranted and the portability claims drop out of the verification scope. The revisit, if F2's cost ever looks wrong, is build-vs-nothing — not build-vs-buy.

F2 re-entry

Evidence trigger, not a date

The F1 pass cheaply logs tagged repeated-pattern rows as it extracts facts — observation only, no synthesis. When ~5 distinct repeated patterns accumulate, or the owner notices re-teaching the same procedure, the F2 decision reopens. Those rows are exactly the input corpus F2 needs on day one, so the deferral is cheap to reverse.

The spec's original tell — build if you want to understand the "improves while you're not looking" property, buy if you only want the property — resolves to build: P1–P3 plus curated memory already yields a persistent, proactive, accreting agent on owned infra, and the one genuinely open-ended piece is deferred until daily use produces evidence it would pay for itself.

Same test, same verdict — cc-connect (C1)

Decision (2026-07-09): rejected; the channel layer is built native. An earlier draft counted cc-connect as an installed bridge. Inspection corrected two things at once: it isn't installed here, and it isn't a bridge — it's a full third-party harness (github.com/chenhg5/cc-connect: drives 10+ agents itself via ACP, owns session management, persistent memory, NL-cron scheduling, and chat-controlled permission modes). There is no pure-adapter mode to demote it into, so adopting it means a resident third-party process that drives agents, holds every channel token, sees all traffic, and keeps its own session/memory/schedule truth models — the Hermes objection with more surface. Forking to salvage its platform adapters inherits a large codebase's update stream for the 2–3 adapters actually needed. Building thin native connectors (discordgo, slack-go) costs days, and puts trust/dedup_key stamping in owned code by construction. cc-connect is retained as reference code for platform quirks — the same posture as reading OpenClaw for design ideas without running it.

§11

Open questions & risks

  • Concurrency. handled by design — the daemon's per-thread goroutine + queue over the durable events table (§3, §4.1, §8) serializes turns, and the one_live_session partial unique index (§6) backstops the router invariant in schema. Becomes a risk only if the implementation drifts back to per-event dispatch.
  • Session scope trap. handled by design — --resume lookups are scoped to the project dir + worktrees, so a session resumed from the wrong cwd silently fails. Every session records its spawn dir in sessions.cwd (§6) and the daemon asserts it on every resume (assistant sessions: ~/approach; workers: the repo path). The failure mode returns if a worker's repo moves or a worktree is pruned — assert, don't assume; when the assert (or the resume itself) fails, §4.6 degrades it to auto-rotate + fact-seeded fresh session + a transparency note, never a hard error.
  • Bare mode drift. --bare (faster, slated to become the -p default) skips hooks/skills/MCP/CLAUDE.md discovery — the opposite of what this harness needs. Pin the non-bare path explicitly.
  • Memory as attack surface. A curated fact written from injected content persists and re-enters future context. Mitigation is structural: fact writes are a gated verb in the policy matrix (§7), and the assembler excludes untrusted-derived rows from context assembly until they're acknowledged in an owner digest — the trust split applied at read time (§5). Residual risk is closed mechanically, not by prompt: the daemon stamps each extracted fact's provenance as the max-untrust over the source log_entries range (§5), so untrusted text in a log cannot talk the reflector into labelling its fact owner-trusted — the model's claimed provenance is never trusted.
  • Skill-synthesis over-fitting. deferred with F2 — without the review gate, the agent teaches itself bad habits from lucky runs. When F2 lands (§10), the gate is not optional — and it must be cross-model (§5): a same-family self-review shares the blind spots it's supposed to catch.
  • Codex sandbox pin. The second engine's hooks are not this harness's hooks — none of the daemon's policy machinery governs it, so its safety here rests entirely on the read-only sandbox flags pinned in the critic skill (§7). One edited flag reopens everything — treat the skill file as security-sensitive config and test the pin, don't assume it.
  • Approval fatigue. If §4.4 fires too often, you'll start rubber-stamping "approve 42" — at which point the gate is theater. Watch the pending_approvals rate in the turns data; if a verb is approved >90% of the time from the owner surface, promote it to allow deliberately rather than letting habituation do it.
  • Noun-partition drift (beads). Two truth models coexist by design (§6): facts in approach.db, work in beads. The partition erodes one convenience at a time — a "fact" that's really a TODO, an issue used to store a durable preference. Make the reflector police it: part of the nightly pass is moving misfiled rows to the right store.
  • Issues as persisted injections. A bd row created from untrusted content is an instruction that persists and re-enters via bd ready — strictly worse than a poisoned fact. The matrix (§7) denies untrusted task writes and provenance-tags gated ones; workers should treat issue bodies from non-owner provenance as data, same as any fetched content.
  • Dependency velocity (beads). Young, fast-moving project — pin the version, wrap bd calls in one skill (a single choke point when flags change), and keep .beads/ in the nightly backup (§6) so a bad upgrade rolls back with the rest of tier-1.
  • Runaway workers. A worker session is an unattended agent with a goal — the costliest failure shape in the system. --max-turns per worker, a per-issue cost ceiling from the turns table, and at most N concurrent workers (start with N=1) in approach.toml.
